# 老爸下象棋 迭代计划与任务拆解

## 1. 文档目标

本文档用于把 [PRD](PRD.md) 和 [系统设计方案](system-design.md) 转化为可执行的迭代节奏、任务拆解和交付标准，避免需求、设计、实现和测试脱节。

## 2. 迭代方法

### 2.1 基本原则

- 先 PRD，后设计，再实现
- 每个迭代必须有明确范围，不做无边界“顺手优化”
- 规则与 AI 类改动必须配套测试
- 用户反馈能复现时，应优先沉淀为测试样例或日志点
- 每次发布必须能从 GitHub Release 下载 APK

### 2.2 标准流程

1. 在 PRD 中确认需求目标、验收口径和优先级
2. 在系统设计中明确模块边界、数据流和风险点
3. 在本文件拆成任务清单、交付件和质量门槛
4. 实现时按任务顺序推进，避免多线并发导致回归失控
5. 完成后补测试、跑 CI、发布 Release APK
6. 将线上或真机反馈回写到 PRD / 系统设计 / 测试集

## 3. 当前基线

当前版本已经具备：

- 单机中国象棋完整基础对弈
- 14 寸平板横屏优先布局
- 悔棋、新局、提示、自动保存
- 双方落子音效、AI 语音播报、通知
- GitHub Actions 自动构建并发布 Release APK
- 一批基础规则测试和 AI 重复着法修正

当前主要短板：

- AI 的收官能力和优势转换仍需继续加强
- 长将 / 长捉 / 追吃循环仍需更严谨的规则化裁决
- 系统 TTS 的可用性观测与自然度仍不稳定
- 老人友好 UI 还缺更体系化的可读性规范
- 测试覆盖率和性能基线还不够成体系

## 4. 版本路线图

### 4.1 V1.1 规则正确性与求胜能力

目标：

- 先把“下得对”打牢，再继续拔高“下得强”

范围：

- 完善被将军后的合法着法过滤
- 强化长将 / 长捉 / 追吃循环识别
- 提升大优局、残局局的求胜策略
- 增加典型残局和规则回归测试

交付件：

- 新增规则回归测试集
- 新增 AI 收官场景测试集
- 规则与 AI 设计更新
- 可安装的 Release APK

完成标准：

- 不再出现”被将军还能乱走别子”的已知规则漏洞
- 不再出现明显追将、追吃无效循环后草率判和
- 高手模式在典型大优残局中能给出更合理的求胜走法

度量指标：

- 规则回归测试数量 ≥ 10 个
- 残局求胜测试覆盖车炮马兵 ≥ 4 种典型局面
- 全部回归测试 100% 通过

### 4.2 V1.2 语音链路与老人友好体验

目标：

- 让反馈链路更稳定、更自然，减少打扰感

范围：

- 重构 TTS 主链路与降级链路
- 增加 TTS 初始化、播报成功、播报失败日志
- 优化播报文案长度与触发条件
- 补齐胜负语音与文字提示一致性
- 统一颜色、字号、按钮密度和状态栏信息设计

交付件：

- 音频与语音链路设计更新
- UI 样式规范补充
- 真机回归清单
- 可安装的 Release APK

完成标准：

- 系统 TTS 可用时优先使用系统语音
- TTS 失败不会静默，日志可定位
- 播报不冗长、不重复打扰
- 老年用户在大屏上可清晰识别棋盘、状态和操作入口

度量指标：

- TTS 初始化和播报日志覆盖率 100%（每次调用均有成功/失败记录）
- UI 字号 ≥ 18sp，触控区域 ≥ 48dp
- 横屏 14 寸平板手工回归清单全部通过

### 4.3 V1.3 AI 性能、测试和工程化

目标：

- 让 AI 和工程质量进入可持续优化状态

范围：

- 建立搜索性能基线
- 增加典型局面 benchmark
- 梳理 AI 参数、时间预算和节点预算策略
- 完善 CI 检查项和发布稳定性
- 补齐日志导出和问题复现说明

交付件：

- AI 性能基线文档
- 测试样例库
- CI 质量门槛清单
- 发布回归清单

完成标准：

- 不同难度下有稳定可预期的响应时间
- 关键规则、关键残局、关键发布流程都有回归兜底
- 新问题能通过日志和测试快速复现

度量指标：

- 入门模式 AI 平均响应 ≤ 500ms，中级 ≤ 1.5s，高手 ≤ 3s
- benchmark 测试套件覆盖 ≥ 5 个典型局面
- CI 构建+测试+发布成功率 ≥ 95%

## 5. 工作流任务拆解

### 5.1 工作流 A：规则引擎

任务列表：

- 梳理将军、应将、对脸将、禁手相关规则口径
- 建立规则缺陷台账，按“必现 / 偶现 / 未证实”分类
- 对每个已确认规则漏洞补最小复现局面
- 将复现局面固化为单元测试
- 修复后回写系统设计中的规则说明

完成定义：

- 漏洞有对应测试
- 修复不会破坏既有规则回归

### 5.2 工作流 B：AI 搜索与评估

任务列表：

- 区分开局、中局、残局的评估目标
- 建立优势转换、压缩将门、限制机动的评估项
- 区分“合理将军”和“无效追将”
- 区分“可得子追吃”和“无效追吃”
- 为典型大优残局建立专项搜索或排序策略
- 对高手模式建立时间和节点上限保护

完成定义：

- 高手模式不会明显卡死
- 典型大优局不会轻易走向低质量和棋
- 每次 AI 改动至少新增一个测试或 benchmark

### 5.3 工作流 C：音频、语音与通知

任务列表：

- 明确系统 TTS 为主链路、内置片段为降级链路
- 给 TTS 初始化和 `speak()` 增加状态观测
- 建立文案规则：短句、少废话、动作明确
- 明确炮吃子与普通吃子的不同播报口径
- 核对通知、界面提示、语音播报三者语义一致

完成定义：

- 用户能判断当前到底是“系统 TTS 失败”还是“应用逻辑未触发”
- 语音问题可通过导出日志定位

### 5.4 工作流 D：老人友好 UI

任务列表：

- 定义棋盘区、状态区、控制区的空间优先级
- 定义字号、触控热区、对比度、按钮命名规范
- 收敛非核心控件数量，避免界面过杂
- 建立横屏 14 寸平板的手工验收清单
- 为胜负结束、AI 思考中、恢复存档等状态设计明确提示

完成定义：

- 不依赖教程也能理解主要操作
- 关键状态始终可见
- 不因背景和棋盘色接近而影响识别

### 5.5 工作流 E：测试与发布

任务列表：

- 维护规则回归测试集
- 增加残局和 AI 行为回归测试集
- 维护 Release APK 安装回归清单
- 确保 GitHub Actions 每次成功后都发布到 Release 页面
- 记录每次发布的已知问题和验证结果

完成定义：

- `testDebugUnitTest` 稳定通过
- `assembleRelease` 稳定通过
- Release 页面始终能下载到最新 APK

## 6. 优先级模型

- `P0`：规则错误、崩溃、无法安装、无法开局、无法发布 APK
- `P1`：AI 明显弱智行为、语音链路失效、重要 UI 可读性问题
- `P2`：体验优化、自然度提升、次要视觉与交互打磨

处理原则：

- 先修 `P0`
- `P1` 按用户感知强度和复现稳定性排序
- `P2` 只能在不影响主线质量的前提下并入

## 7. 任务模板

每个任务建议统一包含以下字段：

- 背景
- 问题描述
- 复现步骤
- 预期行为
- 当前行为
- 涉及模块
- 风险点
- 验收标准
- 对应测试
- 发布说明

## 8. Definition of Ready

一个任务进入开发前，至少满足：

- 需求边界明确
- 验收标准明确
- 影响模块明确
- 是否需要补测试明确
- 是否需要更新文档明确

## 9. Definition of Done

一个任务关闭前，至少满足：

- 代码实现完成
- 自测完成
- 相关测试已补或已更新
- 相关文档已回写
- GitHub Actions 通过
- Release APK 可下载

## 10. 当前建议的下一批任务

建议按下面顺序推进：

1. 把长将 / 长捉 / 无效追吃从“经验惩罚”升级为“规则化裁决模块”
2. 为典型大优残局建立固定测试样例，强化 AI 收官能力
3. 把系统 TTS 成功 / 失败链路彻底观测化，减少静默失败
4. 补一版老人友好 UI 规范，统一颜色、字号、状态区和按钮策略
5. 为 AI 搜索建立基线测试，避免后续优化把性能拖垮

## 11. 发布回滚方案

- GitHub Releases 保留所有历史版本 APK，用户可随时下载旧版本
- 发现严重回归时，优先 revert 问题提交并重新发布，而非在问题基础上继续修复
- 每次发布前在主分支打 tag（格式 `v1.x.y`），确保可精确回溯
- 如果新版存档格式与旧版不兼容，必须在发布说明中标注，并确保旧存档不会导致新版崩溃

## 12. 文档维护要求

- 新需求先更新 [PRD](PRD.md)
- 系统性改动先更新 [系统设计方案](system-design.md)
- 版本节奏、任务拆解和执行顺序更新本文件
- 每次重要发布后，应补一段“已完成 / 未完成 / 已知问题”
